    Hyperparameter Policy For HateXplain Experiments
    ================================================

    Goal
    ----

    We are comparing method performance AND efficiency on HateXplain.

    Different methods CAN use different hyperparameters. That is fair because
    TF-IDF, Bi-LSTM, full fine-tuning, frozen-backbone training, LP-FT, and LoRA
    do not have the same knobs.

    Fairness comes from using the same data, same preprocessing, same validation
    metric, same seed protocol, same search budget rule, and same reporting format.


    Fixed For Everyone
    ------------------

    Everyone must use:

    - Dataset: Hate-speech-CNERG/hatexplain
    - Splits: official train / validation / test
    - Text: text = " ".join(example["post_tokens"])
    - Label: strict majority vote from the 3 annotators
    - Main training labels: 0 = hatespeech, 1 = normal, 2 = offensive
    - Drop no-majority samples for main experiments
    - Model selection metric: validation macro-F1
    - Test set: use only once after hyperparameters are chosen
    - Final seeds: 42, 43, 44

    For Transformer methods:

    - Base model: distilbert-base-uncased
    - Tokenizer: matching DistilBERT tokenizer
    - max_length: 128
    - Padding: dynamic padding with DataCollatorWithPadding


    What Can And Cannot Be Tuned
    ----------------------------

    The short rule:

    - You CAN tune method-specific knobs.
    - You CANNOT tune shared experiment rules.
    - You CANNOT use the test set to choose anything.
    - You CANNOT choose the best seed.

    Do not tune these:

    - dataset choice
    - train / validation / test split
    - text preprocessing
    - label policy
    - label mapping
    - test set usage
    - final seed list
    - main selection metric
    - Transformer base model for main experiments
    - Transformer max_length
    - whether rationales / metadata are included in model input

    If someone changes one of these, they are no longer running the same experiment.


    What Each Method Can Tune
    -------------------------

    TF-IDF + Logistic Regression:

    - ngram_range
    - min_df
    - max_features
    - logistic regression C
    - class_weight

    Bi-LSTM:

    - embedding size
    - hidden size
    - dropout
    - learning rate
    - batch size
    - epochs

    Full DistilBERT fine-tuning:

    - learning rate
    - batch size
    - epochs
    - weight decay
    - warmup ratio

    Frozen DistilBERT:

    - classifier-head learning rate
    - batch size
    - epochs
    - dropout
    - weight decay

    Partial DistilBERT fine-tuning:

    - learning rate
    - number of unfrozen layers
    - batch size
    - epochs
    - weight decay

    LoRA:

    - learning rate
    - LoRA rank r
    - LoRA alpha
    - LoRA dropout
    - target modules
    - epochs

    LP-FT / two-stage methods:

    - stage 1 head learning rate and epochs
    - stage 2 fine-tuning learning rate and epochs
    - weight decay


Better Budget Rule: GPU Hours First
-----------------------------------

The old "8 trials per method" rule is easy, but GPU-hours is better for our
project.

Why:

- One TF-IDF trial and one LP-FT trial do not cost the same.
- Our abstract is about performance AND efficiency.
- Dodge et al. (2019) recommend reporting validation performance together with
  computation budget, such as number of trials or training time.
- Colab GPU type and runtime can change, so we should log the actual hardware
  and time.

So our main policy should be:

Use a fixed HPO time budget per method, plus a maximum trial cap.

This means:

- A method can run random configs until it uses its time budget.
- A method cannot run unlimited cheap trials.
- A very slow method is not punished by being forced to finish the same number
  of trials as a very cheap method.
- We still report the actual number of trials, because time budget alone is not
  enough for reproducibility.


Where The Budget Numbers Come From
----------------------------------

The budget numbers below are not from one paper. They are a practical Colab
decision based on three constraints:

1. Random search is a good default under small budgets.
   Bergstra and Bengio (2012) showed random search is usually more efficient
   than grid search when only some hyperparameters matter.

    2. We only have Colab.
       Colab GPU availability and runtime are not guaranteed. So we need a small,
       fixed budget before experiments start.

    3. Our abstract compares many methods.
       If we tune each method too much, the project becomes too large. We need
       enough trials to avoid a totally unfair default, but not so many that nobody
       can finish.

3. Our abstract compares many methods.
   If each method gets a huge sweep, the project becomes too large.

Therefore, use:

- fixed GPU-hour budget as the main fairness rule
- max trial cap so cheap configs cannot run forever
- final 3-seed evaluation for real claims


Colab Search Budget
-------------------

    Because we only use Colab, do not run huge sweeps.

    Use random search, not full grid search.

Recommended HPO budget:

- TF-IDF + Logistic Regression:
  CPU-only, 30 minutes max, 20 trials max.

- Bi-LSTM:
  1 GPU-hour max, 8 trials max.

- Frozen DistilBERT:
  1 GPU-hour max, 8 trials max.

- Partial fine-tuning:
  2 GPU-hours max, 8 trials max.

- Full fine-tuning:
  2 GPU-hours max, 8 trials max.

- LoRA:
  2 GPU-hours max, 8 trials max.

- LP-FT:
  2 GPU-hours max, 6 trials max.

- Efficient head fine-tuning:
  2 GPU-hours max, 6 trials max.

- Randomly initialized DistilBERT/BERT:
  2 GPU-hours max, 4 trials max.

If a method cannot finish this budget on Colab, record the smaller budget and
say so in the report.

What counts as HPO time?

- Time spent training/evaluating random hyperparameter configs.
- Do not count smoke tests.
- Do not count final 3-seed test runs.
- Do count failed/OOM trials, because they used compute.

Important:

- GPU-hours are only comparable when GPU type is logged.
- A100-hour is not the same as T4-hour.
- If possible, run final timing on the same GPU type.
- If not possible, report GPU type next to every time number.

Simple example:

- Full fine-tuning gets 2 GPU-hours and 8 trials max.
- If it finishes 8 trials in 90 minutes, stop at 8 trials.
- If it only finishes 5 trials in 2 hours, stop at 5 trials.
- Report: "Full FT HPO budget = 2 GPU-hours, completed 5/8 trials on T4."

If the team wants the simplest possible rule, use this instead:

- 8 random-search trials for every method
- 3 final seeds for every method

That is simpler, but less aligned with our performance-vs-efficiency research
question.


    Seed Policy
    -----------

    Do not choose the best seed.

    Recommended workflow:

    1. Run each hyperparameter config once with seed 42.
    2. Pick the top 2 configs by validation macro-F1.
    3. Rerun those top 2 configs with seeds 42, 43, 44.
    4. Choose the config with the best average validation macro-F1.
    5. Run final test evaluation with seeds 42, 43, 44.
    6. Report mean +/- std.

    If Colab time is too tight:

    1. Tune with seed 42 only.
    2. Still run the final selected config with seeds 42, 43, 44.
    3. Write that hyperparameters were selected with one seed due to compute limits.


    Random Hyperparameter Sampling
    ------------------------------

    Do not hand-pick configs after seeing results.

    Before running a method:

    1. Write the search space.
    2. Set one HPO sampling seed, for example:
       hpo_seed = 2026
    3. Randomly sample the planned number of configs.
    4. Save every sampled config.
    5. Run them in order.

    Use different seeds for different jobs:

    - hpo_seed:
      controls which hyperparameter configs are sampled.

    - training_seed:
      controls model initialization, dropout, and data order.

    How to sample:

    - For categorical choices, sample uniformly from the list.
      Example: batch_size from [8, 16].

    - For learning rates and regularization strength, sample on log scale.
      Example: learning_rate from 1e-5 to 5e-5 should be log-uniform, not normal
      uniform.

    - For LoRA alpha, either sample from [8, 16, 32] or set it based on rank:
      alpha = 2 * r.

    Simple Python example:

        import math
        import random

        rng = random.Random(2026)

        def choice(values):
            return rng.choice(values)

        def log_uniform(low, high):
            return 10 ** rng.uniform(math.log10(low), math.log10(high))

        config = {
            "learning_rate": log_uniform(1e-5, 5e-5),
            "batch_size": choice([8, 16]),
            "epochs": choice([2, 3, 4]),
            "weight_decay": choice([0.0, 0.01]),
            "warmup_ratio": choice([0.0, 0.06, 0.1]),
        }

    For W&B Sweeps, use method: random and define the same search space there.
    Still set a run cap. Do not let the sweep run forever.


    Model Initialization
    --------------------

    For pretrained DistilBERT methods:

    - Load weights from distilbert-base-uncased.
    - The classification head is randomly initialized.
    - training_seed controls that random head initialization.

    For frozen backbone:

    - Freeze the DistilBERT backbone.
    - Train only the classifier head.

    For partial fine-tuning:

    - Freeze lower layers.
    - Unfreeze only the selected top layers and classifier head.

    For full fine-tuning:

    - Train all DistilBERT weights.

    For LoRA:

    - Keep the base DistilBERT weights frozen.
    - Train LoRA parameters plus the classifier head.

    For randomly initialized DistilBERT/BERT:

    - Do not load pretrained weights.
    - Initialize the model from config.
    - training_seed controls all model weights.
    - This is expensive and mainly shows why pretraining matters.


    Final Reporting
    ---------------

    For every method, report:

    - validation macro-F1 used for selection
    - test macro-F1 mean +/- std
    - test precision mean +/- std
    - test recall mean +/- std
    - test accuracy mean +/- std
    - trainable parameter count
    - total parameter count
    - training time
    - peak GPU memory
    - GPU type
    - number of tuning trials
    - failed / OOM trials, if any

    If two methods run on different GPU types, say that clearly.


    W&B Logging
    -----------

    Every run should log:

    - method
    - search_stage: smoke / tuning / final
    - trial_id
    - seed
    - dataset
    - data_fraction
    - effective_train_fraction
    - model_name
    - tokenizer_name
    - max_length
    - all hyperparameters
    - validation macro-F1
    - test metrics, only for final runs
    - training time
    - peak GPU memory
    - trainable parameters
    - GPU type
    - git commit hash, if available


    One-Sentence Paper Version
    --------------------------

    Hyperparameters were allowed to differ across methods because each method has
    different optimization controls. For fairness, all methods used the same
    HateXplain splits, preprocessing, validation macro-F1 selection metric, seed
    protocol, and pre-declared Colab search budget. Final test results were reported
    as mean +/- standard deviation across three seeds, together with training time,
    memory, and trainable parameter count.


    Why This Policy
    ---------------

    - Random search is efficient for small HPO budgets (Bergstra and Bengio, 2012).
    - Test data should not be used for hyperparameter selection (Cawley and Talbot,
      2010).
    - NLP model scores can change across random seeds, so use repeated final runs
      (Reimers and Gurevych, 2017; Dodge et al., 2020).
    - Experimental papers should report validation scores, hyperparameter bounds,
      number of runs, runtime, and parameter counts (Dodge et al., 2019;
      ACL-IJCNLP reproducibility checklist, 2021).
    - LoRA and other PEFT methods are designed to reduce trainable parameters, so
      efficiency metrics are part of the comparison (Hu et al., 2021; Houlsby et
      al., 2019).
    - Colab resources are not guaranteed, so we keep the search budget small and
      log GPU type/time/memory (Google Colab FAQ).
